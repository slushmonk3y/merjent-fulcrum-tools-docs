Merjent Fulcrum Tools

A comprehensive JavaScript and Python toolkit for Fulcrum field data collection applications, developed by Merjent environmental consultants.

Notes

  • /report-style contains html and css files related to pdf template design.
  • /tools contains scripts to update Mtools class in all apps and a script to perform bulk replacements of variables/methods - in data event and report scripting environments.
  • build.bat starts shell script /build/buildPkg.ps1 which runs a series of commands to create the modules for distribution.
    • Package is bundled with microbundle.
      • microbundle is a fork of Rollup and modules are written in ESM format. Almost all classes are written in ES6/ES2015 format.
      • The majority of the modules are intended for use in the data events of the Fulcrum App. This is a server-side Node.js environment, therefore the modules located in /dist are bundled as CommonJS modules using module.exports = ... instead of require.
    • buildPkg.ps1 runs microbundle and also uses npm packages rexreplace for text search/replace and terser for mangler/compressor/minification.
    • JSDOC constructed using foodoc.

Overview

Merjent Fulcrum Tools provides a suite of utilities for building and maintaining Fulcrum mobile data collection apps. The package includes:

  • JavaScript Libraries - Reusable classes and helpers for Fulcrum data events and PDF reports
  • Python Scripts - Batch operations, bulk updates, and administrative tools
  • Build Pipeline - Automated bundling, minification, and documentation generation
  • Report Templates - HTML/CSS styling for professional PDF reports

What is Fulcrum?

Fulcrum is a mobile data collection platform for field work. This toolkit extends Fulcrum's capabilities with:

  • Pre-built data validation and field calculations
  • Automated status workflows and permission management
  • Professional PDF report generation
  • Bulk operations across multiple apps

Installation

Prerequisites

  • Node.js 12.0.0 or higher
  • Python 3.x (for Python tools)
  • Fulcrum API Token (for deployment)

Setup

  1. Clone the repository:

    git clone https://github.com/slushmonk3y/merjent-fulcrum-tools.git
cd merjent-fulcrum-tools

  2. Install Node.js dependencies:

    npm install

  3. Install Python dependencies:

    pip install fulcrum-python requests

  4. Configure Fulcrum API:

    • Create tools/core/fulcrum_client.py with your API token
    • See tools/README.md for configuration details

Quick Start

Building the JavaScript Modules

Windows:

build.bat

PowerShell (direct):

powershell -NoProfile -ExecutionPolicy Unrestricted -Command .\build\buildPkg.ps1

Development mode (watch and rebuild):

npm run dev

This generates minified modules in dist/:

  • merjent-tools-de.mmin.js - Data events module
  • merjent-tools-mapp.mmin.js - Data events with MerjentApp
  • merjent-tools-report.mmin.js - PDF reports module
  • merjent-tools-utils.mmin.js - Standalone utilities

Using in Fulcrum

In a Data Event:

//#region !START MERJENT FULCRUM TOOLS!
// ... paste merjent-tools-de.mmin.js or merjent-tools-mapp.mmin.js ...
//#endregion !END MERJENT FULCRUM TOOLS!

// Your custom code
const mtools = new Mtools($form);

ON('load-record', async () => {
  // Use Mtools helpers
  const fieldValue = mtools.getVal('field_name');

  // Use MApp if using merjent-tools-mapp.mmin.js
  const mapp = new MApp($form);
  await mapp.setPermissions();
});

In a PDF Report:

//#region !START MERJENT FULCRUM TOOLS!
// ... paste merjent-tools-report.mmin.js ...
//#endregion !END MERJENT FULCRUM TOOLS!

// Your custom code
const report = new ReportHelpers(CONFIG, data);
const html = report.buildPhotoGrid(photos, 2);

Deploying Updates

Update all apps with latest MTools:

cd tools
python updateMtools.py

Update with search/replace:

python updateMtools.py --search "oldFunc" --replace "newFunc"

Build System

The build pipeline uses:

  • microbundle - Module bundling (Rollup fork)
  • terser - Minification with name preservation
  • rexreplace - Text transformations
  • JSDoc + foodoc - Documentation generation

Build Process

  1. Source transformation - Flattens multiline template strings
  2. Bundling - Creates 4 CommonJS modules for Node.js
  3. Post-processing - Injects version/timestamp, fixes exports
  4. Minification - Creates .mmin.js versions (preserves class/function names for debugging)
  5. Documentation - Generates HTML docs from JSDoc comments

Build Outputs

Each module is compiled as CommonJS for Fulcrum's Node.js server-side runtime:

File Description Exports
merjent-tools-de.mmin.js Data events Mtools, MLoadRecords
merjent-tools-mapp.mmin.js Data events + MApp Mtools, MApp, MLoadRecords
merjent-tools-report.mmin.js PDF reports ReportHelpers, SecOpt
merjent-tools-utils.mmin.js Standalone utilities Utils

Version Management

Version and build timestamp are:

  • Stored in package.json (version, since)
  • Injected into all dist files during build
  • Used by Python tools for deployment tracking

Python Tools

The tools/ directory contains Python scripts for batch operations across all Fulcrum apps.

Architecture

All tools inherit from BaseMToolTool for consistent structure:

  • Centralized Fulcrum API client
  • Reusable app looping infrastructure
  • Standardized CLI argument parsing
  • Progress tracking and result reporting

Key Tools

updateMtools.py

Update MTools code across all apps with optional search/replace:

# Basic update
python updateMtools.py

# With search/replace
python updateMtools.py --search "oldFunc" --replace "newFunc"

# Regex replacement on reports only
python updateMtools.py \
  --search "console\.log\(" --replace "// console.log(" \
  --regex --apply-to reports

See tools/README.md for complete documentation.

Contributing

Development Workflow

  1. Make changes in src/
  2. Build with build.bat or npm run dev (watch mode)
  3. Test in Fulcrum environment
  4. Generate docs with npm run generate-docs
  5. Deploy with python tools/updateMtools.py

Code Style

  • Use ES6+ syntax for source files
  • Document all public methods with JSDoc
  • Preserve existing naming conventions
  • Test changes in actual Fulcrum apps before deploying
